# homelab-config

Config for my homelab Kubernetes cluster. Flux watches this repo and reconciles the cluster state to match what's here.

## Structure

```
├── flux-system/        generated by flux bootstrap, do not edit manually
├── kustomization.yaml  root entrypoint — lists sources, infrastructure, apps
├── sources.yaml        registers the homelab-secrets GitRepository
├── infrastructure.yaml Flux Kustomizations for infra (secrets → controllers → configs)
├── apps.yaml           Flux Kustomization for apps
├── infrastructure/     cluster infrastructure (operators, controllers)
└── apps/               workloads running on the cluster
```

## How it works

Flux runs in the cluster and polls this repo every hour. When it detects a change it applies the diff to the cluster — no manual `kubectl apply` needed.

The apply order is enforced via `dependsOn`:

```
infra-secrets (from homelab-secrets repo)
    └── infra-controllers
            └── infra-configs
                    └── apps
```

Each layer waits for the previous one to be healthy before applying. This ensures secrets exist before controllers start, and CRDs exist before configs use them.

## infrastructure/

Split into two layers:

- **controllers/** — Helm releases for operators that add new capabilities to the cluster (installs CRDs, runs controllers). Current controllers:
  - **tailscale-operator** — exposes services on the tailnet, handles TLS via Tailscale
- **configs/** — resources that _use_ the controllers above, applied after controllers are healthy. Currently empty.

### Adding a new controller

1. Create a directory under `infrastructure/controllers/`:
   ```
   infrastructure/controllers/my-operator/
   ├── helmrepository.yaml
   ├── helmrelease.yaml
   └── kustomization.yaml
   ```
2. Add it to `infrastructure/controllers/kustomization.yaml`
3. If it needs secrets, add the encrypted file to `homelab-secrets` under the same path first

## apps/

Each subdirectory is a workload. See `apps/` for descriptions of each.

### Adding an app

1. Create a directory under `apps/` with `namespace.yaml`, `deployment.yaml`, `service.yaml`, and `kustomization.yaml`
2. Add the directory to `apps/kustomization.yaml`

There are two ways to expose an app on the tailnet:

- **HTTP only** — set `spec.type: LoadBalancer` and `spec.loadBalancerClass: tailscale` on the Service. The operator joins a tailnet node that forwards TCP to the Service. Reachable at `http://<service-name>`. Used by `hello-world`.
- **HTTPS with TLS** — keep the Service as `ClusterIP` and add an `Ingress` with `ingressClassName: tailscale` and `spec.tls.hosts: [<short-name>]` (short hostname only — the operator constructs the FQDN). The operator joins a tailnet node that terminates TLS using a Let's Encrypt cert. Reachable at `https://<short-name>.<tailnet>.ts.net`. Requires HTTPS + MagicDNS enabled in the Tailscale admin console. Add `tailscale.com/http-endpoint: enabled` to also advertise port 80. Used by `whoson`.

## Companion repos

- (private) **[homelab-secrets](https://github.com/adityarathod/homelab-secrets)** — SOPS-encrypted Kubernetes secrets, applied by Flux before anything else

## Bootstrap

```sh
# Install Flux, pointing it at this repo
flux bootstrap github \
  --owner=adityarathod \
  --repository=homelab-config \
  --path=. \
  --personal

# Load the age key so Flux can decrypt secrets
kubectl create secret generic sops-age \
  -n flux-system \
  --from-file=age.agekey=age-key.txt

# Give Flux credentials to clone the private secrets repo
flux create secret git homelab-secrets-auth \
  -n flux-system \
  --url=https://github.com/adityarathod/homelab-secrets \
  --username=adityarathod \
  --password=<github-pat>
```
